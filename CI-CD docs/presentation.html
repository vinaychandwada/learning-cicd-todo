The problem

Manual deployment is painful

Without automation, every release looks like this:

❌ SSH into the server

Type the password, hope you're on the right box.

❌ git pull + npm install + npm run build

Repeated for every service. Wait, watch, hope nothing breaks.

❌ Manually restart processes

Forget to do it? App still runs old code. Restart the wrong one? Outage.

❌ Broken builds reach production

Nothing stopped them. No tests ran. You find out from users.

1
Foundation

Verify your tools

Why: A pipeline is just shell commands. If node, npm, git, or mongod aren't on your machine, the pipeline can't run them either.

# Each should print a version number.
node --version
npm --version
git --version
mongod --version

Required: Node 18+, MongoDB 6+, Git 2.30+.

2
Git setup

Configure your identity

Why: Every commit records who made it. Without this, git commit refuses to run, and GitHub can't link commits to your profile.

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

The email must match one registered on your GitHub account — otherwise commits show as a stranger.

3
Critical

Create a root .gitignore

Why: A .env file committed to Git is a permanent leak. Even deleting it later, it stays in history forever. node_modules/ is also hundreds of MB of regeneratable junk.

Always ignore:

  • node_modules/
  • dist/
  • .env and .env.local
  • *.log
  • actions-runner/

Safe to commit:

  • .env.example (placeholders, no secrets)
  • Source code
  • package.json + package-lock.json
  • Build configs
4
Git

Initialise the repository

Why: Until you run git init, Git has no idea this folder exists. After this, every change is trackable, undoable, and shareable.

git init -b main             # create repo, branch = main
git add .                    # stage all (gitignored files skipped)
git status                   # sanity check — no .env, no node_modules
git commit -m "Initial commit"

Three states of every file in Git:

Working dir
→
Staged
→
Committed
5
GitHub

Push to GitHub

1. Create repo on GitHub

  • New repository
  • Leave all "Initialize with..." boxes unchecked
  • Copy the HTTPS URL

2. Generate Personal Access Token

  • Settings → Developer settings → Tokens (classic)
  • Scopes: repo + workflow
  • Copy once — never visible again

3. Connect & push

git remote add origin \
  https://github.com/you/repo.git

git push -u origin main
# username: your GitHub login
# password: paste the PAT (not your account password)
6
Database

Confirm MongoDB is running

Why: Installed ≠ running. The backend exits immediately if it can't reach Mongo on port 27017.

systemctl status mongod              # is it running?
sudo systemctl start mongod          # start if needed
sudo systemctl enable mongod         # auto-start on boot

Look for two signals in systemctl status:

  • Loaded: ... enabled — will auto-start on boot
  • Active: active (running) — running right now

Secrets

Store secrets in GitHub

Why: We never commit .env to Git. But the pipeline needs those values to deploy. GitHub Secrets = encrypted env vars, auto-masked in logs.

NameValuePurpose
MONGO_URImongodb://127.0.0.1:27017/...DB connection
JWT_SECRETopenssl rand -hex 32Sign login tokens
JWT_EXPIRES_IN1dToken lifetime
BACKEND_PORT5050API port

Once saved, values are write-only — even you can't read them back. That's by design.

11
Automation

The deploy script

deploy/deploy.sh runs on every push. Plain bash — portable to any CI system.

#!/usr/bin/env bash
set -euo pipefail            # exit on any error

# 1. Write .env from secrets passed in by the workflow
cat > node/.env <<EOF
MONGO_URI=${MONGO_URI}
JWT_SECRET=${JWT_SECRET}
EOF

# 2. Build both apps
(cd node  && npm ci && npm run build)
(cd react && npm ci && npm run build)

# 3. Reload PM2
pm2 startOrReload deploy/ecosystem.config.cjs --update-env
pm2 save

Don't forget: chmod +x deploy/deploy.sh

12
The pipeline

GitHub Actions workflow

.github/workflows/ci-cd.yml — two jobs, one fails-fast pattern.

on:
  push:
    branches: [main]

jobs:
  build-test:
    runs-on: ubuntu-latest      # cloud VM, free
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run typecheck && npm run build

  deploy:
    needs: build-test            # only runs if build-test passes
    runs-on: [self-hosted, Linux, X64]
    steps:
      - uses: actions/checkout@v4
      - env:
          MONGO_URI: ${{ secrets.MONGO_URI }}
          JWT_SECRET: ${{ secrets.JWT_SECRET }}
        run: bash deploy/deploy.sh

14
Sanity check

Verify the deployed apps

pm2 list                              # both apps online?

curl http://localhost:5050/           # {"message":"Todo API is running"}
curl -I http://localhost:5173/        # HTTP/1.1 200 OK

Then open http://localhost:5173/ in a browser — your app loads, sign up, create a todo. End-to-end working.

If pm2 list shows status=online but PID=N/A → PM2 couldn't find a binary. Fix the script: path to an absolute path.

15
Resilience

Survive reboots

Without this, rebooting your laptop = all apps dead. PM2 ships a helper that registers it as a systemd service.

pm2 startup
# PM2 prints a "sudo env PATH=... pm2 startup systemd ..." line.
# Copy that exact line and run it.

pm2 save        # freeze current process list for resurrect

Now the boot order is:

Boot
→
systemd
→
MongoDB
→
Runner
→
PM2 → apps
